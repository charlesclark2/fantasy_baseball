# Edge Program — Per-Story Prompts (standalone sessions)

**What this is:** one **`▶ Story prompt`** per *not-yet-completed* story across every epic, so any Trello card can be pulled and run in a fresh Claude Code session on its own (per the guide §0.4 convention). Companion to `edge_program_implementation_guide.md` (the stories' full tasks/AC live there) + `edge_program_technical_spec.md` (design) + `sport_data_platform.md`.
**Last updated:** 2026-06-18 _(refresh on any material change)_

**How to use:** copy a story's block into a fresh session. Blocks are **compact by design** — they inherit shared context by reference (§ = sections of the implementation guide). Every block assumes the **§0.1 conventions**: `dbtf` (not `dbt`); Snowflake via the Snowflake MCP, fully-qualified, no `USE`; never a live Snowflake query on a request path; `uv run python`; hand >1-min scripts to the operator; **do not git commit/push**; market-blind for non-market models (Principle 3); honest-framing for anything user-facing (no win-rate/edge claims). **CI gates BEFORE handoff (never hand off red code): Python → `uv run pytest` (Unit Tests CI); dbt → `dbtf build --select state:modified+` AND `dbtf compile` (both dbt-Build CI jobs) — all green first.** **Every session (model track included) ENDS with an `⏭️ Operator handoff`: the CI-gate result + run-order commands + a copy-pasteable `git add <paths>` of every changed/created file to commit (code, dbt, `sub_model_registry.yaml`, `ablation_results/*.md`, doc edits) + what NOT to commit (artifacts → S3/registry) — so the repo never drifts out of sync.**

**App stories — one fresh session per story:** run app work **one story at a time, each in its own clean session.** For any **[App]** story: open a fresh session, paste the **application-session bootstrap prompt** (persisted at **`app_session_bootstrap.md`**), then paste **exactly one** story block — build, then **end with the `⏭️ Operator handoff` checklist** (Lambda rebuild/deploy, dbt commands, scripts to run, the `git add` for files to deploy, changelog line, verify-after-deploy) so the operator knows exactly what to run. Do **not** chain multiple stories through one long-lived session. App surfaces marked **⏳ §0.3** are **not hand-authored here** — their prompt is *generated by the upstream model session as its final step* (§0.3), filled with the real served PG contract, then run as a normal (bootstrap + that block) app session.

> **🚨 APP TARGET (every `[App]`/`[App/serving]` block) — two live, one dead:** UI → **`frontend/` (Next.js) ONLY**; backend/API → **`app/backend/` (live FastAPI)**; ⛔ **legacy Streamlit UI (`app/streamlit_app.py`, `app/home.py`, `app/pages/`, `app/utils/`) — DO NOT EDIT.** Footgun: `app/` is half-alive (`app/backend/` is live; the rest is dead Streamlit), and Next.js has its own `frontend/app/`. First action in any app session: `cat frontend/package.json` to confirm Next.js; if doing UI work and you open `app/home.py` / `app/pages/*.py`, **STOP — wrong place.** (Full rule: guide §0.2.) A prior session edited the Streamlit UI by mistake (2026-06-18) — this guard prevents a repeat. **Changelog rule: any change the end user would notice ends with a `frontend/data/changelog.json` entry — under the MONDAY of the current week (weeks group Monday→Monday; append to that Monday's block, don't key by the literal ship date).**

**Status legend:** ⬜ open · 🔄 in progress · 🔒 gated · ✅ done (no prompt needed).
